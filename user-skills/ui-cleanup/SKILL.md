## UI 정리해

미푸시 커밋 또는 언스테이징 변경 파일 중 **UI 관련 파일**(`client/src/` 하위 `.tsx`, `.ts`, `.css`)을 대상으로 UI 코드 품질을 점검하고 수정합니다.
사용자가 "전체"라고 명시하면 `client/src/` 전체를 대상으로 합니다.

---

### 1단계: 대상 파일 식별

```bash
# 미푸시 커밋의 변경 파일
git diff --name-only @{upstream}..HEAD 2>/dev/null || git diff --name-only origin/$(git rev-parse --abbrev-ref HEAD)..HEAD 2>/dev/null

# 언스테이징/스테이징 변경 파일
git diff --name-only
git diff --name-only --cached

# 언트래킹 파일
git ls-files --others --exclude-standard
```

- 위 결과를 합쳐 **중복 제거** → `client/src/` 하위 `.tsx`, `.ts`, `.css` 파일만 필터
- `dist/`, `node_modules/`, `.test.`, `.spec.`, `__tests__/` 제외
- 사용자가 **"전체"**를 명시한 경우: `client/src/` 하위 전체를 대상으로 함

---

### 2단계: 전체 파일 읽기 + 분석

대상 파일을 **모두 읽고** 아래 8가지 관점으로 분석합니다.
Agent 도구를 활용하여 병렬로 분석하세요.

#### 체크 항목

| # | 카테고리 | 체크 내용 |
|---|----------|----------|
| 1 | **Raw HTML 제거** | `<button`, `<input`, `<select`, `<textarea`가 직접 사용된 곳 → 공통 컴포넌트(Button, Input, Select, Textarea, ToggleButton, IconButton) 사용 여부 확인. 단, **leaf wrapper 컴포넌트 자체**(Button.tsx 내부 등)는 제외 |
| 2 | **하드코딩 색상** | hex(`#xxx`), `rgb()`, `hsl()`, `rgba()` 값이 직접 사용된 곳 → CSS 변수(`var(--color-*)`) 또는 상수 파일로 전환 필요 여부 확인. 단, SVG 상수 파일(logoColors.ts 등)이나 테마 프리뷰 목적 색상은 상수 객체 정리만 확인 |
| 3 | **컴포넌트 크기** | 200줄 초과 파일 → 디렉토리 구조(`index.tsx` + 서브 컴포넌트)로 분해 가능 여부 확인 |
| 4 | **중복 패턴** | 동일/유사 JSX 블록, 스타일 패턴, 유틸리티 로직이 2곳 이상 반복 → 공통 컴포넌트 또는 유틸리티 추출 가능 여부 |
| 5 | **인라인 서브컴포넌트** | 다른 컴포넌트 파일 내에 정의된 헬퍼 컴포넌트 → 별도 파일로 추출 필요 여부 |
| 6 | **Barrel Export 정합성** | 서브디렉토리의 `index.ts`에서 모든 컴포넌트가 re-export 되는지, 루트 `common/index.ts`와 동기화되어 있는지 확인 |
| 7 | **디자인 토큰 준수** | 매직넘버 간격/크기, `style={{}}` 인라인 스타일 중 CSS 클래스로 대체 가능한 것, Tailwind 유틸리티 미사용 |
| 8 | **재사용성** | 비슷한 UI가 2회 이상 나타나는데 컴포넌트화 되지 않은 것, Props 인터페이스 누락, children 패턴 미활용 |

---

### 3단계: 발견사항 보고

분석 결과를 **카테고리별로 요약**하여 사용자에게 보고합니다.

#### 보고 형식

```
## UI 정리 분석 결과

대상: {N}개 파일 (미푸시 커밋 {M}개 + 언스테이징 {K}개)

### 발견 사항 ({총 건수}건)

#### 1. Raw HTML ({N}건)
- `파일:라인` — 설명

#### 2. 하드코딩 색상 ({N}건)
- `파일:라인` — 설명

#### 3. 대형 파일 ({N}건)
- `파일` ({줄}줄) — 분해 제안

#### 4. 중복 패턴 ({N}건)
- `패턴 설명` — 발생 위치들

#### 5. 인라인 서브컴포넌트 ({N}건)
- `파일:라인` — 컴포넌트명

#### 6. Barrel Export ({N}건)
- `파일` — 누락 항목

#### 7. 디자인 토큰 ({N}건)
- `파일:라인` — 설명

#### 8. 재사용성 ({N}건)
- `패턴 설명` — 추출 제안
```

- 발견 건수가 0인 카테고리는 생략
- 각 항목에 **심각도** 표시: `[must]` 반드시 수정, `[should]` 권장, `[nice]` 선택

---

### 4단계: 수정 수행

- 보고 후 **사용자 확인 없이** 바로 수정을 진행
- 단, 파일 구조 변경(디렉토리 분해, 새 컴포넌트 생성)이 필요한 `[must]` 이슈가 있으면 사용자에게 알리고 방향을 확인
- 수정 원칙:
  - **기능 변경 없이** UI 코드 품질만 개선 (렌더링 결과가 바뀌면 안 됨)
  - Raw HTML → 공통 컴포넌트 교체 시 기존 className/style을 공통 컴포넌트의 props + override 클래스로 매핑
  - 하드코딩 색상 → `var(--color-*)` 또는 상수 파일 추출
  - 중복 패턴 → 유틸리티 함수 또는 공통 컴포넌트 추출
  - Barrel export 누락 → 해당 `index.ts`에 추가
- 수정 완료 후 빌드 검증: `npm run build`

---

### 5단계: 수정 요약

```
## UI 정리 완료

수정: {N}건 / 스킵: {M}건
빌드: ✅ 성공

### 수정 내역
- `파일` — 변경 내용 요약
```

---

### 주의사항

- **기능 변경 금지** — 렌더링 결과가 달라지는 수정은 하지 않음
- **과도한 추상화 금지** — 1회만 사용되는 패턴을 컴포넌트로 추출하지 않음. leaf wrapper(Button 내부의 `<button>` 등)는 정상
- **UiCatalog 제외** — 쇼케이스 페이지는 다양한 컴포넌트를 의도적으로 나열하므로 중복/raw HTML 검사 대상에서 제외
- **커밋하지 않음** — 수정만 하고 커밋은 사용자가 별도로 요청
- **대상 범위** — "전체"를 명시하지 않는 한 미푸시/언스테이지 파일만 대상
