## UI 정리

프론트엔드 소스 중 변경된 UI 파일을 대상으로 UI 코드 품질을 점검하고 수정한다.
"전체"라고 명시하면 프론트엔드 소스 전체를 대상으로 한다.

---

### 1단계: 프론트엔드 소스 감지

프로젝트에서 프론트엔드 소스 디렉토리를 자동 감지한다.

```bash
ls -d frontend/src client/src src/app src/pages src/components 2>/dev/null
```

프론트엔드가 없는 프로젝트 → "프론트엔드 소스를 찾을 수 없습니다" 출력 후 종료.

---

### 2단계: 대상 파일 식별

```bash
git diff --name-only @{upstream}..HEAD 2>/dev/null
git diff --name-only
git diff --name-only --cached
git ls-files --others --exclude-standard
```

- 감지된 프론트엔드 디렉토리 하위 `.tsx`, `.jsx`, `.ts`, `.css`, `.scss` 파일만 필터
- `dist/`, `node_modules/`, `.test.`, `.spec.`, `__tests__/`, `__stories__/` 제외

---

### 3단계: UI 특화 분석

대상 파일을 읽고 아래 관점으로 분석한다. Agent를 활용하여 병렬 분석하면 효율적.

| # | 카테고리 | 체크 내용 |
|---|----------|----------|
| 1 | **Raw HTML** | `<button`, `<input`, `<select` 등 네이티브 요소 직접 사용 → 프로젝트 공통 컴포넌트가 있으면 그걸 사용해야 함. 공통 컴포넌트 래퍼 내부는 제외 |
| 2 | **하드코딩 색상** | hex, rgb, rgba, hsl 값 직접 사용 → CSS 변수 또는 테마 토큰으로 전환. SVG 상수나 테마 프리뷰는 제외 |
| 3 | **인라인 스타일** | `style={{}}` 중 CSS 클래스로 대체 가능한 것 → 클래스 또는 유틸리티로 이동 |
| 4 | **컴포넌트 크기** | 200줄 초과 컴포넌트 → 서브 컴포넌트 분해 제안 |
| 5 | **매직 넘버** | 간격/크기에 숫자 직접 사용 (padding: 24px) → 디자인 토큰 또는 변수 사용 |
| 6 | **재사용성** | 동일/유사 JSX 블록 2곳 이상 반복 → 공통 컴포넌트 추출 제안. 1회만 사용하는 패턴은 제외 |

**프레임워크별 추가 체크:**
- **React**: key prop 누락, useEffect 의존성, 불필요한 리렌더링 (인라인 함수/객체 prop)
- **Vue**: v-for에 key 누락, computed 미사용으로 불필요한 재계산
- **Tailwind 프로젝트**: 커스텀 CSS 대신 유틸리티 클래스 사용 가능한 곳
- **CSS Modules/Styled Components**: 미사용 스타일 정의

---

### 4단계: 발견사항 보고

```
## UI 정리 분석 결과

프론트엔드: {감지된 디렉토리}
대상: {N}개 파일

### 발견 사항 ({총 건수}건)

#### 1. Raw HTML ({N}건)
- `파일:라인` — 설명 [must|should|nice]
...

(발견 건수 0인 카테고리는 생략)
```

---

### 5단계: 수정

- 보고 후 바로 수정 진행
- `[must]` 중 파일 구조 변경이 필요하면 사용자에게 먼저 확인
- **렌더링 결과가 바뀌면 안 됨** — 기능 변경 없이 코드 품질만 개선
- 수정 후 빌드 검증 (감지된 빌드 명령 사용)

---

### 6단계: 수정 요약

```
## UI 정리 완료

수정: {N}건 / 스킵: {M}건
빌드: ✅ 성공

### 수정 내역
- `파일` — 변경 내용 요약
```

---

### 주의사항

- 과도한 추상화 금지 — 1회만 사용하는 패턴을 컴포넌트로 추출하지 않음
- 커밋하지 않음 — 수정만 하고 커밋은 사용자가 별도로 요청
- 쇼케이스/스토리북 파일은 의도적 나열이므로 중복 검사에서 제외
